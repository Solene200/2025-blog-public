# Zustand 与 SWR：这个项目为什么这样做状态管理

这份文档专门回答下面这些问题：

- `Zustand` 是什么，解决什么问题
- `SWR` 是什么，解决什么问题
- 为什么这个项目把编辑态、认证态、预览态、布局态放进 `Zustand`
- 为什么博客索引、分类这类数据交给 `SWR`
- 这样设计带来的好处是什么
- 这样设计的成本和风险是什么
- 如果以后继续扩展功能，什么该放 `Zustand`，什么该放 `SWR`

---

## 1. 先给结论

这个项目的状态管理思路不是“只选一个库统一解决所有问题”，而是把两类问题拆开：

1. 本地 UI / 会话状态，交给 `Zustand`
2. 远端读取的数据，交给 `SWR`

对应到这个项目：

- `Zustand` 管编辑器草稿、图片列表、封面、发布中状态、预览开关、认证状态、首页布局编辑状态、站点配置预览状态
- `SWR` 管博客索引、分类、点赞数这类“从远端拉取并缓存”的数据

这套拆法的核心原因是：

- 编辑器里的输入、拖拽、预览切换会非常频繁，不应该每次都触发请求
- 博客索引、分类这类数据天然是“远端资源”，需要缓存、去重、重验证
- 这两类状态的生命周期完全不同，混在一起会让代码边界变差

可以先把它理解成一句话：

```text
Zustand 管“我现在正在做什么”
SWR 管“服务器上现在有什么”
```

---

## 2. 这个项目里的状态分层

如果把这个仓库按“状态从哪里来、谁负责变化”来分层，大致是这样：

```text
GitHub / public 文件
  └─ 最终持久化数据
       │
       ▼
SWR
  └─ 远端数据缓存、去重、重验证
       │
       ▼
Zustand
  └─ 跨组件共享的本地 UI / 会话状态 / 编辑草稿
       │
       ▼
组件局部 useState
  └─ 只在单个组件内部短暂存在的临时输入态
```

这四层不要混：

- 持久化数据：最终写进仓库或从仓库读取
- SWR：面向“获取远端数据”
- Zustand：面向“共享本地状态”
- `useState`：面向“只在当前组件内部有效的小状态”

这份代码里，其实已经在这么做：

- 博客内容和配置最终存回 GitHub
- 博客索引和分类通过 `SWR` 取
- 编辑器草稿、预览态、认证态、布局态通过 `Zustand` 管
- 配置弹窗内部表单草稿，先用局部 `useState` 管，预览或保存时再回写全局

最后这点很重要，因为它说明这个项目不是“所有状态都塞进 Zustand”，而是在做分层。

---

## 3. Zustand 是什么

`Zustand` 是一个轻量级状态管理库。它的特点是：

- 不需要复杂样板代码
- 不需要像传统全局状态方案那样配置很多 reducer / action
- 不依赖层层 `props` 传递
- 可以把状态放在 React 组件树外部，由多个组件直接共享

最直观的理解是：

```text
组件本地 useState：这份状态只属于当前组件
Zustand store：这份状态属于“当前页面/会话中的多个组件”
```

在这个项目里，`Zustand` 主要负责下面几类状态。

### 3.1 编辑态：`src/app/write/stores/write-store.ts`

这个 store 管理了写文章时最核心的草稿状态：

- 当前是新建还是编辑
- 原始 `slug`
- 表单内容 `form`
- 图片列表 `images`
- 封面 `cover`
- 发布中的 `loading`
- 加载已有博客进入编辑模式的逻辑
- 重置编辑器的逻辑

它对应的消费组件很多：

- `src/app/write/components/editor.tsx`
- `src/app/write/components/sections/meta-section.tsx`
- `src/app/write/components/sections/images-section.tsx`
- `src/app/write/components/actions.tsx`
- `src/app/write/components/preview.tsx`
- `src/app/write/page.tsx`
- `src/app/write/[slug]/page.tsx`

这类状态非常适合放 `Zustand`，因为：

- 标题、正文、摘要、分类、日期都可能频繁更新
- 多个兄弟组件要共享同一份草稿
- 预览页和编辑页需要看到同一份数据
- 导入图片和封面后，需要整个编辑流程都能访问到

如果不用 `Zustand`，就只能：

- 把状态提升到更高层父组件
- 然后把大量 `props` 一层层往下传

这样会让编辑器页面的组件耦合明显升高。

### 3.2 预览态：`src/app/write/stores/preview-store.ts`

预览态只做一件事：

- `isPreview`

以及三类切换动作：

- `openPreview`
- `closePreview`
- `togglePreview`

虽然它很小，但仍然适合单独拆成一个 store。原因是它是“跨组件共享的页面模式”，不是某个按钮自己的局部状态。

`src/app/write/page.tsx` 和 `src/app/write/[slug]/page.tsx` 会根据它决定显示编辑器还是预览页。

### 3.3 认证态：`src/hooks/use-auth.ts`

认证态 store 管理的是：

- 当前是否已认证 `isAuth`
- 当前私钥 `privateKey`
- 设置私钥
- 清理认证
- 刷新认证状态
- 获取 token

这也适合放 `Zustand`，因为认证信息不是只给一个组件用的：

- 博客发布页要用
- 博客列表批量编辑要用
- 图片页、项目页、关于页、分享页、博主页都可能要用

也就是说，它是典型的“会话级共享状态”。

### 3.4 布局编辑态：`src/app/(home)/stores/layout-edit-store.ts`

这个 store 负责：

- 是否处于首页布局编辑模式
- 编辑开始时的快照 `snapshot`
- 取消编辑时回滚
- 拖拽偏移和尺寸调整

这种状态的特点是：

- 只在前端页面里存在
- 改动非常频繁
- 需要多个组件联动
- 用户取消时要能回退

这也是非常典型的 `Zustand` 场景。

### 3.5 配置预览态：`src/app/(home)/stores/config-store.ts`

这个 store 管理：

- `siteContent`
- `cardStyles`
- `regenerateKey`
- `configDialogOpen`

它让首页上的多个卡片组件、背景组件、按钮组件都能读到统一配置。

不过这里有个重要细节：

配置弹窗 `src/app/(home)/config-dialog/index.tsx` 没有把所有输入过程都直接写回 `Zustand`。它先把草稿放到组件局部状态里：

- `formData`
- `cardStylesData`
- 各类上传中的文件状态

只有在“预览”或“保存”时才写回全局 store。

这说明这套设计并不是“看见共享就上 Zustand”，而是会进一步区分：

- 只在弹窗里暂存的输入，先留在组件内部
- 需要影响整个首页预览的状态，再回写全局

这是合理的，因为如果弹窗里的每次输入都实时写全局，会导致主页很多组件跟着重渲染。

### 3.6 已读文章态：`src/hooks/use-read-articles.ts`

这个 store 还用了 `persist` 中间件，把已读文章信息保存在本地存储里。

这是个很好的例子：它不是远端数据，也不是短暂 UI 状态，而是“本地持久化的用户偏好/会话痕迹”。

所以它既不适合放 `SWR`，也不需要写回服务端，放 `Zustand + persist` 正合适。

## 4. SWR 是什么

`SWR` 是一个专门解决“远端数据获取”的 React 库。

它的核心能力是：

- 缓存
- 请求去重
- 自动重验证
- 统一的加载/错误状态
- 手动更新缓存 `mutate`

它名字里的 `stale-while-revalidate` 可以这样理解：

1. 先把缓存里的旧数据给你看
2. 同时后台悄悄发请求验证有没有新数据
3. 如果远端有更新，再把 UI 刷新成新数据

和 `Zustand` 不一样，`SWR` 不是拿来存“我现在输入了什么”的，而是拿来管“远端资源目前是什么内容”的。

---

## 5. 这个项目里 SWR 管什么

### 5.1 博客索引：`src/hooks/use-blog-index.ts`

这个 hook 通过 `useSWR('/blogs/index.json', fetcher)` 获取博客索引。

它做了几件事：

- 拉取 `public/blogs/index.json`
- 统一提供 `items / loading / error`
- 关闭聚焦时自动重验证
- 允许断网恢复后重验证
- 根据认证态过滤隐藏文章

这里 `SWR` 的价值是：

- 同一个页面里多处读取索引时，不需要重复写取数逻辑
- 不用手动维护缓存
- 不用自己处理“请求发没发过”
- 不用每个组件都各自 `fetch`

### 5.2 分类：`src/hooks/use-categories.ts`

这个 hook 通过 `useSWR('/blogs/categories.json', fetcher)` 获取分类列表。

它和博客索引很像：

- 都是静态 JSON 资源
- 都是远端读
- 都适合缓存
- 都会被多个地方复用

比如博客列表页和写作页元信息区域都可能需要分类。

### 5.3 点赞数：`src/components/like-button.tsx`

点赞按钮是另一个很典型的 `SWR` 场景：

- 先通过 `GET` 取当前点赞数
- 点击后通过 `POST` 更新服务端
- 再通过 `mutate` 直接更新本地缓存值

这说明 `SWR` 不只是“读静态数据”，也可以管理一类轻量的远端计数数据。

不过它仍然是“远端资源状态”，而不是“页面本地草稿状态”。

## 6. 为什么这里不用一个库全包

很多人第一次接触状态管理时，会自然想问：

- 既然 `Zustand` 能存状态，为什么不把博客索引也放进去
- 既然 `SWR` 能缓存，为什么不把编辑器草稿也放进去

这个项目不这么做，是因为这两类状态的变化方式完全不同。

### 6.1 为什么编辑器草稿不用 SWR

编辑器草稿有这些特点：

- 更新频率极高，基本每次输入都变
- 大量状态只是本地临时值
- 不适合每次变动都重验证远端
- 需要跨多个组件共享

如果把它交给 `SWR`，会出现职责错位：

- `SWR` 的优势在远端数据生命周期
- 编辑器草稿根本不是“远端资源”
- 会让“取数逻辑”和“本地输入逻辑”混在一起

### 6.2 为什么博客索引和分类不用 Zustand

博客索引和分类有这些特点：

- 数据源在远端文件
- 需要统一获取
- 需要缓存
- 需要请求去重
- 需要错误和加载状态
- 可能需要重新验证

如果把它们全放进 `Zustand`，你就得自己手写：

- 什么时候请求
- 是否重复请求
- 请求中状态
- 请求失败状态
- 缓存过期策略
- 页面聚焦后要不要刷新
- 断网恢复后要不要刷新

这些本来就是 `SWR` 的强项，没有必要自己重造一层。

---

## 7. 这样设计的直接收益

### 7.1 降低组件耦合

写作相关组件不需要一层层传递大串 `props`：

- 编辑器改正文
- 侧边栏改元信息
- 预览页读草稿
- 操作栏执行发布

它们都直接共享 `useWriteStore`。

这让组件职责更聚焦：

- 编辑器只关心输入
- 侧边栏只关心元数据
- 操作栏只关心发布和预览动作

### 7.2 避免编辑操作触发不必要请求

这是这套设计非常重要的一个收益。

输入标题、拖图片、改摘要、切换预览、调整首页卡片位置，本质上都是本地交互。

这些动作如果混进远端数据流里，就容易导致：

- 无意义重新拉取
- 组件加载状态抖动
- UI 不稳定
- 请求量增加

而现在：

- 本地编辑操作只改 `Zustand`
- 远端索引和分类由 `SWR` 单独控制

两条线互不干扰。

### 7.3 远端数据获取逻辑统一

博客索引和分类都通过专门的 hook 提供给页面：

- `useBlogIndex()`
- `useCategories()`

页面组件不需要关心：

- 请求地址是什么
- 请求失败怎么处理
- 结果结构是否稳定

它们只消费 hook 的结果即可。

### 7.4 缓存和请求去重更自然

如果多个组件都要读 `/blogs/index.json`，使用 `SWR` 时：

- 相同 key 的请求会共享缓存
- 不需要每个组件各发一次
- 不需要自己造一个全局缓存管理器

### 7.5 更适合“草稿 -> 预览 -> 发布”的编辑器流程

这个项目的编辑器不是简单输入框，而是一条完整的工作流：

```text
输入内容
  -> 处理图片
  -> 生成预览
  -> 导入密钥
  -> 提交到 GitHub
```

在这条链路里，草稿状态明显比远端索引更重要。用 `Zustand` 持有这条链路的本地会话状态，是比较顺手的选择。

## 8. 这套设计的弊端和代价

这套设计方向是对的，但不代表它没有成本。这里要把代价说清楚。

### 8.1 Zustand 分片是有了，但有些 store 还是偏大

`useWriteStore` 已经不只是一个“小型编辑 store”，它同时承担了：

- 表单数据
- 图片管理
- 封面管理
- 加载中状态
- 加载已有博客逻辑
- 重置逻辑

这在中小项目里仍然可接受，但继续长下去会有风险：

- store 变得越来越像“小型业务中心”
- 修改一个功能时，容易影响别的功能
- 组件订阅范围一大，就会带来更多重渲染

### 8.2 订阅粒度还可以更细

这个仓库里有不少地方直接把整个 store 解构出来用，例如：

- `useWriteStore()`
- `useConfigStore()`

这会带来一个实际问题：

- 只要 store 里相关字段变化，组件就可能重新渲染

更细的做法通常是 selector 订阅，例如只取：

- `state.form`
- `state.loading`
- `state.configDialogOpen`

这样可以减少与当前组件无关的更新。

### 8.3 store 之间已经开始出现交叉依赖

例如：

- 认证 store 会读取配置 store 的 `isCachePem`
- 布局编辑 store 会直接操作配置 store

这不是错误，但说明全局状态边界开始耦合：

- 某个 store 的行为依赖另一个 store
- 后续扩展时更容易形成复杂联动

如果规模继续增长，维护成本会上升。

### 8.4 SWR 的读取层做了，但写后缓存刷新还不完整

点赞按钮有手动 `mutate`，这是比较完整的：

- 写入成功后，本地缓存马上更新

但博客发布、删除、分类批量修改后，并没有看到对：

- `/blogs/index.json`
- `/blogs/categories.json`

做统一 `mutate` 或失效处理。

当前代码更像“写入 GitHub 后，等部署完成，再刷新页面看结果”。

这会带来两个后果：

1. 逻辑简单，少写很多同步代码
2. 用户体验不够即时，刚发布完时本地列表可能还是旧的

所以这套方案更偏“最终一致性”，不是“前端立即强一致”。

### 8.5 这里的 SWR 更依赖内存缓存，不太依赖浏览器 HTTP 缓存

当前 fetcher 普遍使用了：

```ts
fetch(url, { cache: 'no-store' })
```

这表示：

- 浏览器不会优先用 HTTP cache
- 每次重验证都是真请求
- 缓存更多依赖 SWR 自己维护的内存层

这样做的优点是数据更新更直接，缺点是缓存收益比“浏览器缓存 + SWR”组合更有限。

### 8.6 认证缓存有安全换便利的代价

`useAuthStore` 支持根据配置决定是否缓存 PEM。

这提升了体验：

- 用户不用每次重新导入密钥

但也意味着：

- 本地持久化认证材料会提升暴露面

这不是状态管理库本身的问题，但属于这套设计一起带来的工程权衡。

## 9. 为什么说它“降低组件耦合”

这里的“降低组件耦合”不是说组件之间完全没关系，而是说：

- 组件之间不需要通过父组件串联数据流
- 组件不需要知道彼此内部实现
- 数据共享通过统一 store / hook 完成

以写作页为例：

### 以前一种常见但更耦合的写法

```text
WritePage
  ├─ state: form
  ├─ state: images
  ├─ state: cover
  ├─ state: loading
  ├─ 把 form / setForm 传给 Editor
  ├─ 把 images / cover / updateXxx 传给 Sidebar
  ├─ 把 loading / publish / preview 传给 Actions
  └─ 把 form / cover 传给 Preview
```

这种写法的问题是：

- 父组件会越来越胖
- props 越传越多
- 任意字段调整都会波及很多组件接口

### 现在的写法

```text
WriteEditor      -> useWriteStore()
WriteSidebar     -> useWriteStore()
WriteActions     -> useWriteStore() + usePreviewStore()
WritePreview     -> useWriteStore() + usePreviewStore()
```

这样组件之间共享的是“同一个业务状态源”，而不是彼此直接传数据。

这就是这里说的降低耦合。

## 10. 为什么说它“避免编辑操作触发不必要请求”

这句话要拆开看。

### 10.1 什么叫“不必要请求”

例如用户做了这些操作：

- 输入标题
- 改一段正文
- 粘贴一张图片
- 切换预览
- 拖拽首页卡片
- 切换布局编辑模式

这些操作的本质都是本地交互，并不意味着：

- 博客索引变了
- 分类列表需要重新获取
- 服务器需要重新验证

如果把这些本地状态和远端数据耦合在一起，就容易出现：

- 本地草稿一变就触发远端刷新
- 页面反复进入 loading
- 多个组件重复发请求

### 10.2 现在这套拆分为什么能避免它

因为编辑动作只影响 `Zustand` 里的本地状态：

- `useWriteStore`
- `usePreviewStore`
- `useLayoutEditStore`
- `useAuthStore`

而远端数据只在真正需要时由 `SWR` 管：

- `useBlogIndex`
- `useCategories`
- `LikeButton` 的点赞数请求

所以：

- 写作不影响博客索引请求
- 预览不影响分类请求
- 布局拖拽不影响文章列表请求

这就是“把本地交互态和远端数据态拆开”的直接收益。

---

## 11. 这套设计里一个容易被忽略但很重要的点

不是所有状态都应该进 `Zustand` 或 `SWR`。

配置弹窗就是一个很好的反例。

在 `src/app/(home)/config-dialog/index.tsx` 里，很多输入先存在组件自己的 `useState` 里：

- `formData`
- `cardStylesData`
- 上传文件项

只有预览或保存时，才写回 `useConfigStore`。

这样做的原因是：

- 弹窗里的输入过程不需要全站实时感知
- 如果每次输入都打到全局，会造成更多重渲染
- 局部草稿本来就应该尽量局部化

所以真正合理的设计不是：

```text
所有共享状态都放 Zustand
```

而是：

```text
局部临时状态 -> useState
跨组件共享的本地业务状态 -> Zustand
远端数据获取和缓存 -> SWR
最终持久化 -> GitHub / 文件
```

---

## 12. 以后继续开发时，怎么判断该放哪一层

可以用下面这套简单规则判断。

### 放 `useState`

适合：

- 只在当前组件内部使用
- 生命周期很短
- 不需要给兄弟组件共享

例子：

- 弹窗的 tab 切换
- 输入框聚焦态
- 某个按钮的 hover 辅助状态

### 放 `Zustand`

适合：

- 需要跨多个组件共享
- 是本地 UI / 会话 / 草稿状态
- 更新频繁
- 不应该随着每次变化都请求后端

例子：

- 编辑器草稿
- 预览开关
- 首页布局编辑状态
- 登录 / 认证态
- 本地已读记录

### 放 `SWR`

适合：

- 数据源在远端
- 需要统一拉取
- 希望有缓存、去重、重验证
- 可能会被多个组件共同读取

例子：

- 博客索引
- 分类列表
- 点赞数
- 未来如果有“远端标签列表”“远端用户资料摘要”等，也适合这一层

---

## 13. 对这套实现的总体评价

如果只评价设计方向，这套方案是合理的，而且符合这个项目的实际需求。

因为这个项目不是传统后台管理系统，也不是纯静态展示站，而是一个带轻量内容编辑能力的前端博客：

- 本地交互很重
- 远端读取相对轻
- 写入路径依赖 GitHub API

在这种场景下：

- 用 `Zustand` 承担本地编辑流程
- 用 `SWR` 承担远端读取缓存

是比“全部塞进一个全局 store”更清晰的方案。

不过如果继续迭代，最值得优先优化的是：

1. `Zustand` 订阅粒度更细，少用整块解构
2. 大 store 再拆分，减少职责膨胀
3. 发布、删除、分类修改后补上 `SWR mutate`，让本地列表刷新更及时
4. 控制 store 之间的交叉依赖，避免状态网状耦合

---

## 14. 一句话总结

这套设计本质上是在做一件事：

```text
把“本地高频交互状态”和“远端可缓存数据”分开管理
```

这样做的结果是：

- 编辑器体验更稳定
- 组件之间少传很多 props
- 不会因为本地输入就触发无意义请求
- 远端数据获取逻辑更统一

对应的代价是：

- 需要维护多层状态边界
- store 设计不好时容易膨胀
- 写后缓存同步需要额外补齐

但整体上，这仍然是这个项目里更合适的工程取舍。

## 15. SWR 具体是怎么做到这些的

上面说 `SWR` 负责缓存、请求去重、自动重验证、统一加载状态，这里再往下拆一层，讲它到底是怎么做到的。

先给直接结论：

- 是，`SWR` 已经把大部分机制封装好了
- 业务代码通常只需要提供三样东西：
  - `key`：这份远端数据的身份
  - `fetcher`：怎么把数据取回来
  - `options`：什么时候重验证、多久去重、是否聚焦刷新等策略

最常见的调用方式就是这个形状：

```ts
const { data, error, isLoading, mutate } = useSWR(key, fetcher, options)
```

在这个项目里，对应的实际例子有：

- `useSWR('/blogs/index.json', fetcher, ...)`
- `useSWR('/blogs/categories.json', fetcher, ...)`
- `useSWR(\`${ENDPOINT}?slug=...\`, fetcher, ...)`

也就是说，业务层并没有手写一整套“缓存管理器”，而是把这部分交给了 `SWR`。

### 15.1 `key`：SWR 先靠 key 识别“你到底在取哪份数据”

`SWR` 的第一层核心概念不是请求函数，而是 `key`。

例如：

- `/blogs/index.json`
- `/blogs/categories.json`
- `https://.../api/like?slug=xxx`

对 `SWR` 来说，这个 `key` 就是缓存槽位的名字。只要两个组件用的是同一个 `key`，`SWR` 就会认为它们读的是同一份远端资源。

所以它可以做到：

- 共享缓存
- 共享请求结果
- 共享错误状态
- 共享重新验证结果

这就是为什么说 `SWR` 特别适合“多个组件读同一份远端数据”的场景。

### 15.2 `fetcher`：你负责“怎么取”，SWR 负责“什么时候取、取完怎么分发”

`fetcher` 一般就是一个异步函数，例如：

```ts
const fetcher = async (url: string) => {
  const res = await fetch(url)
  if (!res.ok) throw new Error('Fetch failed')
  return res.json()
}
```

这里的职责分工很清楚：

- 你负责定义怎么发请求、怎么解析响应、什么情况下算失败
- `SWR` 负责决定什么时候触发这个函数，以及拿到结果后怎么更新缓存和通知订阅者

换句话说，`fetcher` 只是“取数实现”，不是“状态管理实现”。

### 15.3 缓存：SWR 内部会按 key 维护一层内存缓存

当你第一次调用：

```ts
useSWR('/blogs/index.json', fetcher)
```

大致会发生这些事：

1. `SWR` 先看内部缓存里有没有 `/blogs/index.json`
2. 如果有缓存，先把缓存值立刻返回给组件
3. 如果策略允许，再后台发起一次重新验证
4. 新结果回来后，更新缓存并通知所有订阅这个 key 的组件重新渲染

也就是说，`SWR` 的“先给旧数据，再悄悄更新”不是一句口号，而是它内部缓存流程的一部分。

这也是 `stale-while-revalidate` 这个名字的来源：

- `stale`：先用旧值
- `revalidate`：后台校验远端有没有新值

### 15.4 请求去重：同一个 key 的并发请求，SWR 会尽量合并

如果两个组件几乎同时调用：

```ts
useSWR('/blogs/index.json', fetcher)
```

没有 `SWR` 时，常见结果是：

- 两个组件各发一遍请求

有 `SWR` 时，它会维护同 key 的“进行中请求”记录。大致逻辑是：

1. 第一个组件发现还没有请求，于是发请求
2. 第二个组件过来时，发现这个 key 已经有请求在路上
3. 第二个组件直接复用同一个请求结果
4. 请求完成后，两个组件一起拿到更新结果

这就是请求去重。

在点赞按钮里还显式设置了：

```ts
dedupingInterval: 1000 * 10
```

意思是：在这个时间窗口内，同 key 的重复请求会更积极地复用结果，而不是反复打远端。

### 15.5 自动重验证：SWR 会在合适的时机主动再拉一次

`SWR` 的另一个核心能力是“重验证”，也就是重新去远端确认数据是否变了。

常见触发时机包括：

- 组件挂载
- 窗口重新获得焦点
- 网络断开后恢复
- 调用 `mutate` 后要求重新拉取

这些行为都可以通过 `options` 调整。

例如这个项目里：

- `revalidateOnFocus: false`
- `revalidateOnReconnect: true`

这代表：

- 用户切回页面时，不自动重新拉
- 断网恢复后，可以重新拉一次

所以 `SWR` 的“自动重验证”不是死板固定的，而是由它的策略配置控制。

### 15.6 统一的加载和错误状态：这也是封装好的一部分

如果手写 `fetch`，你通常还要自己维护：

- `loading`
- `error`
- `data`

而 `SWR` 已经把这层统一成 hook 返回值了：

```ts
const { data, error, isLoading } = useSWR(...)
```

这样页面组件就不用在每个地方都重复写：

- `const [loading, setLoading] = useState(...)`
- `const [error, setError] = useState(...)`
- `useEffect(() => { fetch... })`

这也是它和普通 `fetch + useEffect` 最大的工程差别之一。

### 15.7 `mutate`：手动更新缓存，或者主动要求重新验证

`mutate` 可以理解成“我想手动干预这份 key 对应的缓存状态”。

它常见有两类用途：

#### 第一类：直接改缓存，不立刻重拉

点赞按钮里就是这个模式：

```ts
await mutate(value, { revalidate: false })
```

意思是：

- 我已经知道最新点赞数应该是多少
- 先直接把缓存改成这个值
- 不马上再发请求确认

这样 UI 会立刻更新，看起来更跟手。

#### 第二类：告诉 SWR 这份数据可能过期了，重新拉一次

例如另一种常见用法是：

```ts
mutate()
```

或者全局按 key 触发重新验证。

意思是：

- 当前缓存先别完全信了
- 再去远端拿一遍最新数据

这在“写操作完成后刷新列表”里很常见。

### 15.8 用伪流程看一遍 SWR 的工作过程

把上面的机制合起来，一次 `useSWR` 调用大致可以理解成下面这样：

```text
组件调用 useSWR(key, fetcher, options)
  ├─ 先查 key 对应的内存缓存
  │   ├─ 有缓存：先返回缓存
  │   └─ 没缓存：先返回空值 / loading
  │
  ├─ 判断当前是否需要发请求
  │   ├─ 如果同 key 已有进行中请求：复用它
  │   └─ 否则：执行 fetcher
  │
  ├─ 请求完成后更新 key 对应缓存
  │
  └─ 通知所有订阅这个 key 的组件重新渲染
```

如果后面发生这些事件：

- 页面重新聚焦
- 网络恢复
- 业务代码调用 `mutate`

那 `SWR` 会再按照同样的 key 重新跑一遍“重验证流程”。

### 15.9 它和浏览器 HTTP 缓存不是同一层

这里特别容易混淆。

`SWR` 自己有一层“应用内内存缓存”，而浏览器本身还有一层 HTTP 缓存。这两层不是一回事。

在这个项目里，fetcher 普遍写的是：

```ts
fetch(url, { cache: 'no-store' })
```

这意味着：

- 浏览器 HTTP cache 基本不参与
- `SWR` 主要依赖自己的内存缓存和重验证机制

所以这里说“有缓存”，更多指的是：

- 当前前端运行时里的 `SWR cache`

而不是：

- 浏览器磁盘缓存或 CDN 缓存

### 15.10 为什么说“这些机制大部分已经被封装好了”

因为如果不用 `SWR`，你自己要重复造出这些东西：

- 一张按 URL 或 key 存储结果的缓存表
- 一张记录“哪些请求正在进行中”的表
- 一套加载态和错误态管理
- 一套窗口聚焦 / 网络恢复时的重拉逻辑
- 一套“写后更新缓存”的能力
- 一套通知所有使用者重新渲染的机制

而用了 `SWR` 之后，业务层通常只需要关心：

- 这份数据的 key 是什么
- 请求函数怎么写
- 什么时候该 revalidate
- 写操作完成后要不要 mutate

也就是说，`SWR` 已经把“远端资源状态管理”的大框架封装好了，业务层只需要补业务规则。

### 15.11 放到这个项目里，业务代码到底还需要负责什么

结合当前仓库，业务代码自己负责的主要是：

- 定义 key，例如 `/blogs/index.json`
- 写 fetcher，例如怎么处理 `res.ok`
- 决定是否在未认证时过滤隐藏文章
- 决定哪些场景需要 `revalidateOnFocus`
- 决定写操作后是否 `mutate`

而下面这些能力，是 `SWR` 已经帮你处理掉的：

- 缓存容器
- 同 key 订阅关系
- 同 key 请求去重
- 返回统一的 `data / error / isLoading`
- 重验证的触发框架
- 手动更新缓存接口

所以更准确地说：

```text
SWR 不是帮你写“业务请求”
SWR 是帮你封装“远端数据在前端怎么活着”
```
