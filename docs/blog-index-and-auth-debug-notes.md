# 博客索引丢失与登录失败排查记录

这份文档记录了我这次遇到的两类问题：

1. 新写的博客详情页可以直接访问，但博客索引页没有出现，首页文章卡片也没有出现。
2. 一段时间里始终无法登录，后来发现本机系统时间慢了两分钟，没有校时。

这不是一份“修复执行记录”，而是一份“问题呈现、排查思路、根因判断、代码级解决方法”的分析文档。

---

## 1. 问题一：详情页能打开，但索引页和首页文章卡片没有出现

### 1.1 现象是怎样的

我当时看到的是一个很容易让人误判的组合现象：

- GitHub 仓库里已经有新文章目录和内容
- Vercel 也显示部署成功
- 直接在地址栏输入这篇文章的单页 URL，可以正常打开并渲染
- 但是 `/blog` 文章索引页里没有这篇文章
- 首页文章卡片也没有出现这篇文章

如果只看最后两条，很容易先怀疑：

- `slug` 写错了
- 路由没有生效
- Vercel 缓存没更新
- markdown 或 `config.json` 内容有问题

但我后来判断，真正关键不在 `slug`，而在于“详情页的数据源”和“列表页的数据源”其实不是同一条链路。

### 1.2 为什么我一开始就不认为重点在 `slug`

因为这篇文章的详情页是可以直接访问的。

只要 `/blog/<slug>` 能打开，至少说明下面几件事大概率已经成立：

- 路由参数本身是可用的
- `public/blogs/<slug>/index.md` 存在
- `public/blogs/<slug>/config.json` 至少能被读取
- 前端渲染单篇文章的逻辑没有直接报错

如果 `slug` 真有问题，最典型的结果应该是：

- 详情页直接 404
- 或者详情页提示文章不存在

而不是“详情页能打开，但列表页没有它”。

所以我当时真正要验证的不是 `slug`，而是：

> 单篇详情页和博客列表页，到底是不是依赖同一份数据。

### 1.3 我是怎么拆数据链路的

把页面拆开以后，这个问题就清楚很多。

#### 1.3.1 详情页的读取链路

详情页在 `src/app/blog/[id]/page.tsx` 中，通过 `loadBlog(slug)` 读取文章。

对应读取逻辑在 `src/lib/load-blog.ts`：

- 读 `/blogs/<slug>/config.json`
- 读 `/blogs/<slug>/index.md`

也就是说，详情页是“按 slug 直接去拿这篇文章自己的文件”。

只要文章目录还在，详情页就可能正常打开。

#### 1.3.2 列表页和首页文章卡片的读取链路

博客索引页 `src/app/blog/page.tsx` 和首页文章卡片 `src/app/(home)/aritcle-card.tsx` 都依赖 `useBlogIndex()`。

`useBlogIndex()` 在 `src/hooks/use-blog-index.ts` 里只做一件事：

- 请求 `/blogs/index.json`

首页文章卡片再通过 `useLatestBlog()` 从这份索引里挑一篇最新文章出来展示。

所以：

- 详情页依赖 `public/blogs/<slug>/...`
- 列表页和首页文章卡片依赖 `public/blogs/index.json`

这两条链路不是同一个数据源。

### 1.4 这意味着什么

一旦出现下面这种情况：

- 文章目录已经写进仓库
- 但 `public/blogs/index.json` 没有这篇文章

就会出现一个非常典型的现象：

- 详情页能打开
- 索引页看不到
- 首页文章卡片也看不到

这正好和我遇到的现象完全一致。

### 1.5 我后面拿到的证据

为了避免只靠猜，我继续看了远端提交记录和远端文件状态。

最后确认到几个关键事实：

#### 1.5.1 新文章目录在远端是存在的

远端 `origin/main` 上确实存在：

- `public/blogs/login-mode/index.md`
- `public/blogs/login-mode/config.json`
- 对应的图片资源

而且 `config.json` 里：

- `hidden` 是 `false`

所以这不是“文章被隐藏”导致的。

#### 1.5.2 远端 `index.json` 里没有这篇文章

同一个远端分支上，`public/blogs/index.json` 里没有 `login-mode`。

这一步很关键，因为它直接解释了：

- 为什么 `/blog/login-mode` 可以打开
- 为什么 `/blog` 列表里没有它
- 为什么首页文章卡片也没有它

#### 1.5.3 新增文章时其实一开始是正常的

我继续看 Git 历史后发现：

- `19b19b7` 这次提交是 `新增文章: login-mode`

在这次提交里：

- `public/blogs/login-mode/...` 被新增
- `public/blogs/index.json` 里也已经新增了 `login-mode`

这说明发文那一次本身是成功的。

#### 1.5.4 真正出问题的是后面的“批量保存索引”

紧接着的另一个提交是：

- `27af8fd`
- 提交信息：`删除:5-17,4-5,sdk-fix | 更新索引 | 更新分类`

这个提交删除了一批文章目录，同时重写了 `public/blogs/index.json`。

问题就在这里：

- 它删除了 `5-17`、`4-5`、`sdk-fix`
- 但它也顺带把 `login-mode` 从 `index.json` 里抹掉了
- 可是它并没有删除 `public/blogs/login-mode/` 目录

于是就形成了“文章存在，但索引里消失”的不一致状态。

### 1.6 根因不是发文失败，而是后续索引被旧快照覆盖

问题的核心不在 `pushBlog()`，而在博客管理页的批量保存逻辑。

#### 1.6.1 发文逻辑本身没有明显问题

发文逻辑在 `src/app/write/services/push-blog.ts`。

它在发布文章时会同时写入：

- `public/blogs/<slug>/index.md`
- `public/blogs/<slug>/config.json`
- `public/blogs/index.json`

其中更新索引的逻辑调用的是 `prepareBlogsIndex()`，它会：

1. 先读取远端当前的 `public/blogs/index.json`
2. 按 `slug` 合并新文章
3. 再把新的索引写回去

也就是说，正常发文这一步的模型是“基于远端最新状态做增量合并”。

#### 1.6.2 真正危险的是 `saveBlogEdits()`

博客列表页的编辑保存逻辑在 `src/app/blog/services/save-blog-edits.ts`。

它原本的做法是：

1. 页面先通过 `useBlogIndex()` 拉到一份索引
2. 用户在页面上做删除、分类调整等操作
3. `saveBlogEdits()` 直接拿页面内存中的 `nextItems`
4. 用 `nextItems` 整份重写 `public/blogs/index.json`

问题在于：

- `nextItems` 只是“当前这个页面会话里看到的快照”
- 它不一定等于远端最新索引

如果当前页面拿到的是旧快照，就会出现这种情况：

1. 我打开博客管理页时，页面里还是旧列表
2. 之后我又新增了一篇新文章，远端已经有了
3. 但当前博客管理页没有重新同步到这篇新文章
4. 我再在这个旧页面里做批量删除或分类保存
5. `saveBlogEdits()` 就会把这份旧快照整份写回 `index.json`
6. 远端后来新增的文章，就会被这次保存“覆盖掉”

这就是这次问题的真正根因。

### 1.7 为什么这个问题特别容易发生

除了整份覆盖之外，还有两个前端行为会放大这个问题：

#### 1.7.1 `useBlogIndex()` 用的是 SWR 内存缓存

`src/hooks/use-blog-index.ts` 里：

- 请求使用 `fetch(url, { cache: 'no-store' })`
- 但 `useSWR('/blogs/index.json', ...)` 自己会保留内存缓存

所以这个索引不是“每次都强制以页面打开时远端为准”，而是“同一浏览器会话里会复用 SWR 缓存”。

#### 1.7.2 `revalidateOnFocus` 被关掉了

同一个文件里还设置了：

- `revalidateOnFocus: false`

这意味着：

- 页面切回前台时，不会自动重新拉最新索引

如果用户先打开旧列表，再发文，再回来批量操作，这个页面很可能一直拿着旧数据。

#### 1.7.3 发布和保存后也没有统一 `mutate`

当前代码没有在发文、删除、批量保存后统一刷新：

- `/blogs/index.json`
- `/blogs/categories.json`

所以用户界面更容易停留在旧状态。

### 1.8 这次问题最准确的一句话总结

这不是 `slug` 问题，也不是详情页渲染问题，而是：

> 文章正文目录已经成功发布，但后续一次基于旧页面快照的“索引重写”把这篇文章从 `public/blogs/index.json` 里覆盖掉了。

### 1.9 代码层的解决方法应该怎么写

这次我不直接执行线上修复，只把代码解决方法写清楚。

#### 1.9.1 最核心的修复原则

`saveBlogEdits()` 不应该再直接用页面内存里的 `nextItems` 整份覆盖远端索引。

它应该改成：

1. 先读取远端最新的 `public/blogs/index.json`
2. 只删除用户明确删除的 `slug`
3. 只更新当前页面实际改动过的字段
4. 保留远端最新索引里那些“当前页面没有感知到，但真实存在”的文章
5. 最后再排序并写回去

#### 1.9.2 代码实现思路

可以在 `src/app/blog/services/save-blog-edits.ts` 中增加一个“先读远端索引”的步骤，例如：

```ts
const remoteItems = await readRemoteBlogIndex(token)
const removedSlugSet = new Set(uniqueRemoved)
const nextBySlug = new Map(nextItems.map(item => [item.slug, item]))

const mergedItems = remoteItems
  .filter(item => !removedSlugSet.has(item.slug))
  .map(item => {
    const edited = nextBySlug.get(item.slug)
    if (!edited) return item

    return {
      ...item,
      category: edited.category
    }
  })

const sortedItems = mergedItems.sort((a, b) => new Date(b.date).getTime() - new Date(a.date).getTime())
```

这个改法的关键点是：

- 删除操作只删明确删除的文章
- 分类调整只改已有文章的分类字段
- 远端最新索引里其他未触碰的文章仍然保留

这样就不会因为页面快照旧而误删新文章。

#### 1.9.3 再往前走一步的增强方案

除了上面的最小修复，还可以继续加强：

1. 在发布、删除、批量保存成功后，主动 `mutate('/blogs/index.json')`
2. 同时 `mutate('/blogs/categories.json')`
3. 如果保存前发现远端索引已经变化，可以提示用户“列表已更新，请刷新后再保存”
4. 如果要做得更稳，可以把“页面加载时的索引版本”与“提交前远端索引版本”做比对

#### 1.9.4 为什么这个方案比“怀疑 slug”更重要

因为 `slug` 问题只会影响某一篇的路由命中。

而“索引被旧快照整份覆盖”是更高一级的数据一致性问题：

- 它不仅会影响某一篇文章
- 它还会影响博客列表
- 会影响首页文章卡片
- 会影响后续所有基于索引的功能

所以从工程上讲，这个问题比 `slug` 更值得优先处理。

## 2. 问题二：本机系统时间慢了两分钟，导致一直登录不上

### 2.1 现象是怎样的

我之前还遇到过一个非常隐蔽的问题：

- GitHub App 的登录链路一直过不去
- 私钥、仓库、部署看起来都没问题
- 但就是一直“登录不上”
- 后来发现本机系统时间慢了两分钟，没有自动校时
- 系统时间校准后，登录问题消失

这类问题比博客索引问题更容易误导人，因为它表面上看像是：

- 私钥错了
- App ID 错了
- GitHub App 没安装好
- 仓库权限不对
- 网络请求失败

但这一次最后证明，真正的触发点是本机时钟漂移。

### 2.2 我为什么一开始没有立刻想到“时间”

因为“系统时间慢两分钟”听起来太不像一个前端登录问题了。

直觉上更容易先怀疑这些更显眼的点：

1. 私钥是否导错
2. `APP_ID` 是否填错
3. GitHub App 是否安装到正确仓库
4. 当前项目的 `OWNER / REPO / BRANCH` 是否对
5. GitHub API 是否限流或网络异常

这些检查都很合理，但它们都属于“配置层”。

而这套登录链路里其实还有一层更底层的条件：

> JWT 是否在 GitHub 看来处于合法时间窗口内。

### 2.3 登录链路在代码里是怎样跑的

认证流程主要在两个文件里：

- `src/lib/auth.ts`
- `src/lib/github-client.ts`

顺序大致是：

1. `getAuthToken()` 先看本地缓存里有没有 token
2. 如果没有，就从当前私钥生成 GitHub App JWT
3. 用这个 JWT 去请求 GitHub 安装信息
4. 再换取 installation token

关键函数调用链是：

- `getAuthToken()`
- `signAppJwt()`
- `getInstallationId()`
- `createInstallationToken()`

### 2.4 为什么系统时间会影响这个链路

关键就在 `signAppJwt()`。

这个函数里直接使用了本机当前时间：

```ts
const now = Math.floor(Date.now() / 1000)
const payload = { iat: now - 60, exp: now + 8 * 60, iss: appId }
```

这说明：

- JWT 的签发时间 `iat`
- JWT 的过期时间 `exp`

都不是来自 GitHub 服务器，而是来自**当前这台机器的系统时间**。

只要本机时钟漂移了，签出来的 JWT 时间窗口就会一起漂移。

### 2.5 这为什么会导致“明明配置对了，却还是登不上”

因为 GitHub 在校验 GitHub App JWT 时，不只看签名对不对，还会看这个 token 的时间是否合法。

一旦本机时间和真实时间偏差过大，就可能出现：

- GitHub 认为这个 JWT 还没到生效时间
- 或者认为这个 JWT 已经过期
- 或者直接把它视为无效时间窗口内的令牌

在代码里，这个问题最终会表现成：

- `getInstallationId()` 请求失败
- 常见表现是 401 或和 JWT 无效相关的认证失败信息

所以表面上看是“GitHub App 登录不上”，本质上却可能是：

> 本机用错误的时间签了一个时间窗口不合法的 JWT。

### 2.6 为什么“系统时间只慢两分钟”也值得警惕

因为这条链路的时间窗口本来就不大。

这里的 JWT：

- `iat` 取的是 `now - 60`
- `exp` 取的是 `now + 8 * 60`

它不是一个可以长期容忍大误差的 token。

所以哪怕只有 1 到 2 分钟级别的时钟漂移，也足以让认证结果变得不稳定，尤其是在：

- 请求本来就有延迟
- 本地缓存状态和远端状态交错
- 用户多次重试

这些因素叠加时，问题会更难看出来。

### 2.7 这次问题里，我最后是怎么把它锁定到“系统时间”的

这次最有决定性的证据不是理论推导，而是实际现象闭环：

1. 登录一直失败
2. 代码路径里又确实存在基于本机 `Date.now()` 的 JWT 签发
3. 后来发现系统时间慢了两分钟
4. 校时之后，登录恢复正常

也就是说，在这次环境里：

- “时间偏差存在”
- “认证流程依赖本机时间”
- “校时后问题消失”

这三个条件已经足以把根因锁定到本机时钟漂移。

### 2.8 这个问题除了 JWT，还会影响哪里

除了 JWT 签发，`src/lib/auth.ts` 里本地 token 缓存也依赖 `Date.now()` 判断过期：

```ts
if (expiresAt <= Date.now() + TOKEN_EXPIRY_BUFFER_MS) {
  clearTokenCache()
  return null
}
```

这意味着系统时间不准时，还会让缓存判断变得不可靠：

- 可能把本该失效的 token 继续当成有效
- 也可能把本可用的 token 提前判成失效

所以时钟漂移不只是“签一次 JWT 的问题”，它会影响整条认证体验。

### 2.9 这类问题以后应该怎么更快想到

如果下次再遇到“GitHub App 登录不上”，我会按下面的顺序排查：

1. 私钥是否正确、是否完整
2. `NEXT_PUBLIC_GITHUB_OWNER / REPO / BRANCH / APP_ID` 是否正确
3. GitHub App 是否安装到了目标仓库
4. 仓库权限是否真的给到
5. 浏览器里有没有旧 token / 旧 pem 缓存
6. 本机系统时间是否准确，是否已经校时

这里第 6 条非常容易被忽略，但这次事实证明它必须被纳入第一层排查清单。

## 3. 两个问题放在一起看，我的真正收获是什么

这两次问题表面上完全不同：

- 一次看起来像文章没发布好
- 一次看起来像登录配置错了

但它们有一个共通点：

> 不能只盯着“表面 UI 现象”，要先回到“数据链路和状态来源”。

### 3.1 第一个问题教会我的事

不要一上来就怀疑 `slug`。

更有效的做法是先问：

- 单篇详情页读的是什么
- 列表页读的是什么
- 首页文章卡片读的又是什么

只要把数据源拆开，问题很快就能缩到 `index.json`。

### 3.2 第二个问题教会我的事

不要把“时间”当成和业务无关的系统噪音。

只要链路里涉及：

- JWT
- token 过期
- 本地缓存有效期

那么系统时间就是业务的一部分。

### 3.3 以后再遇到类似问题，我应该先问什么

我以后会先问自己三件事：

1. 这个页面真正依赖的是哪份数据，而不是我以为它依赖哪份数据？
2. 这次写入是“增量合并”，还是“拿旧快照整份覆盖”？
3. 这条认证链路有没有任何一步依赖本机时间？

如果这三件事先问清楚，很多表面看起来“很玄”的问题，其实都会变得很具体。

## 4. 最终结论

### 4.1 关于博客索引问题

这次不是 `slug` 问题。

真正的问题是：

- 新文章目录已经成功发布
- 详情页因此可以直接访问
- 但后续一次博客管理页的批量保存，用旧页面快照整份重写了 `public/blogs/index.json`
- 结果把新文章从索引里覆盖掉了

所以根因是：

> 索引写入策略存在“旧快照覆盖远端最新状态”的数据一致性问题。

### 4.2 关于登录失败问题

这次也不是单纯的 GitHub App 配置错误。

真正的问题是：

- 认证链路里的 JWT 签发依赖本机 `Date.now()`
- 本机系统时间慢了两分钟，没有校时
- 导致 GitHub App 认证时间窗口异常
- 校时后登录恢复正常

所以根因是：

> 本机时钟漂移影响了基于时间窗口的 GitHub App JWT 认证。
